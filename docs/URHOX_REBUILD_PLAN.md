# OpenFoot Manager UrhoX/Lua 重建方案

本文档用于指导将 OpenFoot Manager 从当前的 Rust + React + Tauri 架构，重建为 UrhoX + Lua 单进程游戏项目。重建目标不是逐行迁移代码，而是复用原项目的数据、玩法规则和产品结构，用 Lua 重新实现核心逻辑，并用 UrhoX/urhox-libs/UI 重建竖屏界面。

## 1. 重建目标与边界

### 1.1 目标

- 以 UrhoX 为运行时，Lua 作为主要开发语言。
- UI 采用竖屏优先设计，默认面向手机或竖屏窗口。
- 语言只保留中文，不迁移 i18next 多语言系统。
- 优先完成除比赛引擎外的所有经营、数据、UI、存档、赛季流转模块。
- 保留后续接入完整比赛模拟引擎的接口边界，避免前期系统与临时实现强绑定。

### 1.2 暂不实现内容

- 暂不移植 Rust `engine` 比赛引擎。
- 暂不实现完整逐分钟比赛、实时比赛事件、战术指令影响、半场调整影响、点球大战等复杂模拟。
- 暂不做 React/Tauri 兼容层。
- 暂不保留 SQLite 存档格式。

### 1.3 可复用资产

- `src-tauri/data/default_teams.json` 可作为球队模板来源。
- `src-tauri/data/default_names.json` 可作为姓名池来源。
- 原项目文档中的规则、公式、模块设计可作为 Lua 实现依据。
- 原 UI 的信息架构和页面拆分可复用，但界面需要按竖屏重新设计。
- 中文文本可以直接内置在 Lua 表或 JSON 中，不需要多语言 key。

## 2. 总体架构

UrhoX 版本采用单进程架构：

```text
UrhoX App
├── Lua Runtime
│   ├── app/              # 启动、场景、路由、全局事件
│   ├── core/             # 游戏状态、日期、赛季、回合推进
│   ├── domain/           # 球员、球队、联赛、经理、职员等数据模型
│   ├── systems/          # 训练、财务、合同、转会、新闻、球探等玩法系统
│   ├── data/             # JSON 数据读取、世界生成、静态配置
│   ├── persistence/      # JSON 存档、设置、存档槽
│   ├── ui/               # 竖屏 UI 页面、组件、弹窗
│   └── match/            # 比赛接口与临时占位实现，后续替换为完整引擎
└── Resources/
    ├── Data/             # 球队、姓名、配置 JSON
    ├── Saves/            # 本地 JSON 存档
    ├── Textures/         # 队徽、图标、背景
    └── Fonts/            # 中文字体
```

### 2.1 模块拆分原则

- 每个系统独立一个 Lua 模块，不把所有逻辑堆到单个脚本。
- `domain` 只放数据结构和基础计算，不直接操作 UI。
- `systems` 只修改 `GameState`，不直接创建 UI 控件。
- `ui` 只通过控制器调用系统接口，不直接写复杂业务规则。
- `match` 使用接口隔离，即使当前只做占位，也要保留 `start_match`、`finish_match`、`build_report` 等函数。

## 3. 数据模型重建

### 3.1 GameState

`GameState` 是 UrhoX 版本唯一的运行时根状态，替代原 Rust `Game` + Zustand store + Tauri IPC。

必须包含：

- 当前日期、赛季、回合状态。
- 当前经理 ID、当前执教球队 ID。
- 球队、球员、职员、经理列表。
- 联赛、赛程、积分榜、赛季统计。
- 收件箱消息、新闻文章、世界历史。
- 财务、合同、转会、球探、青训、董事会目标。
- 当前 UI 所需的临时状态，例如当前选中球员、选中球队、当前页签。

### 3.2 球员模型

需要覆盖原项目所有主要字段：

- 身份：ID、比赛名、全名、生日、国籍、足球身份、出生国家。
- 位置：自然位置、当前位置、可客串位置。
- 脚法：惯用脚、逆足能力。
- 属性：速度、体能、力量、敏捷、传球、射门、抢断、盘带、防守、站位、视野、决策、镇定、侵略性、团队合作、领导力、手型、反应、制空。
- 动态状态：状态、士气、长期体能、伤病、退役标记。
- 评分：当前能力、潜力、特性。
- 合同：合同结束日、工资、身价、续约状态。
- 赛季统计：出场、进球、助攻、黄牌、红牌、评分等。
- 生涯记录：效力球队、赛季、出场、进球、荣誉。
- 训练：个人训练重点。
- 转会：挂牌、出租、报价列表。
- 士气核心：比赛时间、合同、表现、球队成绩等影响项。

### 3.3 球队模型

必须包含：

- 基础信息：ID、名称、简称、城市、国家、足球身份、球场、容量、成立年份、颜色。
- 竞技：声望、阵型、打法、首发 11 人、队长/点球/任意球/角球人选、近期状态。
- 财务：余额、工资预算、转会预算、赛季收入、赛季支出、财务流水、赞助、设施。
- 训练：球队训练重点、强度、周计划、训练小组。
- 人员：经理 ID、球员、职员。
- 历史：历年排名、胜平负、进失球。

### 3.4 其他核心模型

- `Manager`：姓名、生日、国籍、当前球队、职业统计、执教履历、声望。
- `Staff`：角色、属性、专长、工资、所属球队。
- `League`：球队列表、赛程、积分榜、当前轮次。
- `Fixture`：主队、客队、日期、状态、比分、进球事件。
- `InboxMessage`：标题、正文、发送人、日期、分类、优先级、动作、上下文引用、已读状态。
- `NewsArticle`：标题、正文、来源、日期、分类、关联球队/球员/比赛、已读状态。
- `WorldHistory`：赛季冠军、奖项、转会记录、经理变动、球队历史。

## 4. 竖屏 UI 重建方案

### 4.1 设计原则

- 默认竖屏比例：推荐 1080x1920 或 720x1280 作为设计基准。
- 单手操作优先：主导航放底部，关键按钮放屏幕下半区。
- 页面使用卡片式信息流，减少横向表格。
- 所有长列表都要支持搜索、筛选、排序和虚拟滚动或分页。
- 数据密集页面采用“摘要卡 + 列表 + 详情抽屉”的结构。
- 弹窗尽量改为底部抽屉，适合竖屏。
- 中文字体必须提前打包，避免系统字体缺失。

### 4.2 主导航

原桌面侧边栏不适合竖屏，改为：

- 底部一级导航：主页、球队、赛事、市场、消息。
- 顶部状态栏：日期、球队、资金、继续按钮。
- 更多菜单：经理、设置、名人堂、全球数据库、存档。

建议一级导航映射：

| 一级入口 | 包含模块 |
| --- | --- |
| 主页 | 总览、下一场、联赛摘要、任务提示、职位机会 |
| 球队 | 阵容、战术、训练、职员、球员详情、球队详情 |
| 赛事 | 赛程、积分榜、赛事、赛季结算、奖项 |
| 市场 | 转会、全球转会中心、球探、青训、球员库、球队库、经理库 |
| 消息 | 收件箱、新闻、董事会、随机事件 |

### 4.3 页面组件规范

- `TopBar`：日期、返回、标题、继续按钮。
- `BottomNav`：五个主入口。
- `Card`：统一圆角、阴影、标题、右侧操作。
- `StatPill`：小型数据标签。
- `PlayerRow`：球员行，展示姓名、位置、能力、状态、士气。
- `TeamRow`：球队行，展示排名、队名、积分、状态。
- `BottomSheet`：球员详情、操作菜单、筛选器。
- `ConfirmDialog`：保存、出售、解雇、退出等危险操作确认。
- `Toast`：轻量反馈。

## 5. 玩法系统重建清单

### 5.1 游戏生命周期

职责：

- 启动游戏。
- 创建经理。
- 生成或载入世界。
- 选择球队。
- 进入 Dashboard。
- 保存、读取、退出到主菜单。

最低实现：

- 新游戏全流程可走通。
- 支持至少 3 个存档槽。
- 自动保存当前进度。

### 5.2 世界生成

职责：

- 读取球队模板和姓名池 JSON。
- 生成球队、球员、职员、经理、联赛和赛程。
- 提供硬编码兜底数据。

必须实现：

- 每队 22 名球员：2 GK、7 DEF、7 MID、6 FWD。
- 每队 4 名职员：助理教练、教练、球探、理疗师。
- 额外生成自由职员。
- 生成双循环联赛赛程。

### 5.3 日期与回合推进

职责：

- 每次点击“继续”推进一天。
- 判断是否比赛日。
- 非比赛日处理训练、恢复、消息、新闻、随机事件。
- 比赛日暂时走占位比赛流程。

比赛引擎未完成前的策略：

- 用户比赛可以显示“比赛功能开发中”，允许跳过并生成临时比分。
- AI 比赛使用极简占位结果：按球队声望、主场优势、随机数生成比分。
- 占位结果只产生比分、积分榜和少量统计，不产生完整逐分钟事件。

### 5.4 联赛与赛程

职责：

- 双循环赛程生成。
- 积分榜更新。
- 比赛状态流转：未开始、进行中、已完成。
- 排序规则：积分、净胜球、进球数。

必须实现：

- 赛程列表。
- 积分榜。
- 比赛日结果写回。
- 赛季结束检测。

### 5.5 阵容系统

职责：

- 球员列表。
- 首发管理。
- 替补和未入选展示。
- 阵容安全检查。
- 位置适配。

必须实现：

- 自动选择首发。
- 手动调整首发。
- 阵容不足时提示。
- 伤病球员不可进入首发。

### 5.6 战术系统

职责：

- 阵型。
- 打法。
- 首发位置。
- 定位球角色。

先期实现：

- 保存阵型、打法、首发和角色。
- 暂不把战术强绑定到完整比赛模拟。
- 占位比赛中可用战术给结果概率提供轻微修正。

### 5.7 训练系统

职责：

- 训练重点：体能、技术、战术、防守、进攻、恢复。
- 训练强度：低、中、高。
- 周计划：密集、均衡、轻量。
- 训练小组。
- 个人训练覆盖。
- 属性成长、状态消耗、恢复。

必须实现：

- 非比赛日训练。
- 年龄、职员、专长影响成长。
- 理疗师影响恢复。
- 低状态消息提醒。

### 5.8 职员系统

职责：

- 职员角色：助理教练、教练、球探、理疗师。
- 职员属性：训练、识人、潜力判断、理疗。
- 专长：体能、技术、战术、防守、进攻、门将、青训。

必须实现：

- 职员影响训练和恢复。
- 职员工资进入财务。

### 5.9 球员特性

职责：

- 按属性自动计算特性。
- 在 UI 展示特性标签。

### 5.10 球员评分与能力

职责：

- 按位置计算当前能力。
- 生成潜力。

## 6. 比赛模块后置方案

虽然暂不实现 Rust 比赛引擎，但必须从第一阶段就定义接口。

### 6.1 MatchAdapter 接口

建议 Lua 接口：

```lua
local MatchAdapter = {}

function MatchAdapter.startMatch(gameState, fixtureId, options)
end

function MatchAdapter.simulateInstant(gameState, fixtureId)
end

function MatchAdapter.finishMatch(gameState, matchReport)
end

function MatchAdapter.buildPlaceholderReport(gameState, fixtureId)
end

return MatchAdapter
```

### 6.2 占位比赛报告

占位报告至少包含：

- fixture_id。
- home_team_id。
- away_team_id。
- home_goals。
- away_goals。
- scorer_events：可为空或用简化射手生成。
- team_stats：射门、控球、犯规可先用随机值。
- player_stats：可先只更新出场、进球、助攻。

### 6.3 后续完整比赛引擎接入点

未来完整引擎只替换 `match/placeholder_engine.lua`，其他系统不应改动：

- 赛程只读取 `MatchReport`。
- 新闻只读取 `MatchReport`。
- 球员统计只读取 `MatchReport`。
- UI 只读取 `MatchSnapshot` 或 `MatchReport`。

## 7. 存档方案

### 7.1 文件格式

SQLite 改为 JSON 文件：

```text
Saves/
├── save_001.json
├── save_002.json
├── save_003.json
├── autosave.json
└── index.json
```

### 7.2 save 文件结构

```json
{
  "version": 1,
  "saved_at": "2026-05-26T19:30:00+08:00",
  "game_state": {},
  "ui_state": {
    "last_screen": "dashboard",
    "last_tab": "home"
  }
}
```

### 7.3 迁移策略

- 每个存档带 `version`。
- 读取时调用 `migrations.lua`。
- 只保证 UrhoX 版本自己的存档向后兼容。
- 不承诺直接读取原 SQLite 存档。

## 11. 风险与应对

### 11.1 UI 信息密度风险

足球经理游戏数据密集，竖屏容易拥挤。

应对：

- 采用摘要卡、详情页、底部抽屉。
- 表格改为卡片列表。
- 高级数据折叠展示。

### 11.2 模块规模风险

原项目模块很多，如果前期不拆分，Lua 项目会快速失控。

应对：

- 严格按 `domain/core/systems/ui` 拆分。
- 每个系统只暴露少量公共函数。
- 所有跨模块修改都通过 `GameState`。

### 11.3 比赛引擎后置风险

没有完整比赛引擎时，部分系统反馈会不真实。

应对：

- 使用 `MatchAdapter` 保持接口。
- 占位比赛只提供最小结果。
- 不在其他系统写死占位逻辑。

### 11.4 存档兼容风险

原项目 SQLite 存档难以直接复用。

应对：

- UrhoX 版从 JSON 存档开始。
- 数据模板可复用，旧存档不作为第一版目标。

### 11.5 中文字体与排版风险

UrhoX 环境下中文字体、换行、字号需要额外验证。

应对：

- 项目初期先完成中文字体加载。
- 所有 UI 组件按中文长度测试。
- 提供大字号设置。

## 12. 第一版验收标准

- 可以查看和管理阵容、战术、训练、职员、财务、合同。

当上述经营模块稳定后，再启动比赛引擎专项：
