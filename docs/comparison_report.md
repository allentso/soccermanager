# OpenFoot Manager - `origin` 与 `new` 项目对比报告

> 对比对象：`origin`（Tauri + React + Rust + SQLite 原始实现）与 `new`（UrhoX + Lua 5.4 重构实现）
> 分析日期：2026-05-27
> 结论口径：本文重点评估 `new` 相对 `origin` 的功能覆盖、扩展价值、当前短板与下一步优先级。

---

## 一、总体结论

`new` 已经从单纯的 Lua 复刻，发展成一个更完整的移动端/轻量端足球经理原型。相比 `origin`，它的经营系统覆盖面更宽：五大联赛世界、欧冠、世界杯、升降级、董事会、声望、求职、奖项、历史、名人堂、随机事件、AI 管理、新闻系统、青训和职员管理都已形成闭环。

不过，`new` 目前最大的短板也很明确：**比赛引擎深度和自动化测试体系弱于 `origin`**。`origin` 的 Rust 引擎已经具备逐分钟、事件驱动、加时赛/点球、体能与 live match 管理等更成熟的基础；`new` 目前仍以 `placeholder_engine.lua` 为核心，虽然 UI 已经覆盖赛前、实况、半场和赛后流程，但底层模拟还没有达到同等策略深度。

| 维度 | `origin` | `new` | 结论 |
|------|----------|-------|------|
| 技术栈 | Tauri + React/TS + Rust + SQLite | UrhoX + Lua 5.4 + JSON 存档 | `new` 更轻量，适合快速迭代；`origin` 工程基础更成熟 |
| 核心经营循环 | 已完整 | 已完整 | 两者都可支撑完整赛季循环 |
| 世界规模 | 俱乐部/联赛/赛事框架较完整 | 五大联赛 + 欧冠 + 世界杯 + 升降级 | `new` 的世界层更扩展 |
| 比赛引擎 | 逐分钟事件驱动，Rust 模块化 | 占位引擎 + 完整比赛 UI | `origin` 引擎更强，`new` UI 参与感更足 |
| 转会/合同/球探 | 买卖、租借、合同、自由球员、球探 | 多轮谈判、AI 买卖、租借、自由球员 FSM、区域球探 | `new` 已基本追平并局部增强 |
| 财务/董事会 | 财务、预算、设施、董事动作 | 财务健康、注资、赞助、商业活动、董事满意度 | `new` 经营反馈更丰富，但设施升级未完成 |
| 球员深度 | 角色、训练、合同、事件 | 个人士气、阵容角色、职业历史、训练分组 | `new` 玩家管理链路更细 |
| UI 覆盖 | React 桌面端页面完整 | 26 个 Lua UI 页面，移动/横屏取向 | `new` 页面覆盖更广，但需要更多交互打磨 |
| 测试保障 | Vitest + Rust tests 覆盖广 | 暂无系统化自动测试 | `new` 回归风险明显更高 |

## 二、架构与工程基础对比

### 2.1 `origin`：成熟桌面应用架构

`origin` 是标准桌面端架构：React/TypeScript 负责 UI，Rust 负责核心逻辑与比赛引擎，SQLite 负责本地持久化。它的优势在于边界清晰、类型系统强、测试覆盖广，适合作为稳定基线。

主要特点：

| 模块 | 实现特点 |
|------|----------|
| 前端 | React + TypeScript + Zustand + i18n，多页面组件化 |
| 后端 | Rust crate 分层，包含 domain、engine、ofm_core、db、Tauri commands |
| 存档 | SQLite + migration，适合长期演进 |
| 测试 | 前端 Vitest、Rust workspace tests、命令层 tests 均有覆盖 |
| 风险 | 桌面端工程成本较高，跨端迁移不如 Lua 轻量 |

### 2.2 `new`：轻量重构 + 系统扩展

`new` 使用 UrhoX + Lua 5.4 重构，整体结构已经按 `app / core / domain / systems / match / persistence / data / ui` 分层。当前约 69 个 Lua 文件，后端系统 21 个，UI 页面 26 个，已经不是 demo 级结构。

主要特点：

| 模块 | 实现特点 |
|------|----------|
| 核心状态 | `game_state.lua` 管理全局状态和序列化 |
| 回合推进 | `turn_processor.lua` 负责每日推进、比赛日分流、训练、新闻、谈判、事件 |
| 系统层 | 21 个 manager 覆盖转会、财务、合同、训练、董事会、士气、AI、奖项等 |
| UI 层 | 26 个 screen 覆盖主菜单、仪表盘、阵容、战术、比赛、转会、青训、名人堂等 |
| 存档 | JSON 多槽 + autosave，开发迭代快 |
| 风险 | 动态语言 + 缺少测试，系统越多越需要回归保护 |

## 三、核心模块对比

| 功能 | `origin` | `new` | 评价 |

| 换人/战术干预 | 已实现 | 已实现 | `new` 已完成旧报告中的核心缺口 |
| 半场调整 | 已实现 | 已实现 | `new` 已完成 |
| 技术统计/赛后评分 | 已实现 | 已实现 | `new` 目前基于占位引擎生成 |

| 战术/属性深度影响 | 较成熟 | 部分影响，仍偏简化 | `new` 需要 B4 引擎补强 |

`new` 的比赛体验层已经明显改善：玩家可以赛前确认、比赛中介入、半场调整、赛后查看报告。它解决的是“玩家参与感”问题。但目前结果生成仍主要依赖 `placeholder_engine.lua`，还没有达到 `origin` Rust engine 那种逐分钟、事件驱动、战术与球员状态深度绑定的程度。

**判断**：比赛 UI 完成度约 80%+；比赛引擎真实深度约 40%。下一阶段最值得投入的是 `match_engine.lua`、`tactics_resolver.lua`、`match_report.lua` 三件套。

## 五、当前主要短板与风险

### 5.1 高优先级短板

| 问题 | 影响 | 建议 |
|------|------|------|
| 比赛引擎仍是占位核心 | 阵容、战术、士气、训练的策略价值无法充分体现 | 优先实现逐分钟 `match_engine.lua` |
| 战术解析不足 | 阵型和打法选择容易变成 UI 开关 | 实现 `tactics_resolver.lua`，把阵型/打法/角色映射到事件概率 |
| 比赛报告深度不足 | 赛后反馈无法解释胜负原因 | 实现 `match_report.lua`，输出关键事件、球员评分、战术反馈 |
| 缺少自动化测试 | 系统变多后回归风险快速上升 | 先为纯逻辑 manager 加最小 Lua 测试/断言脚本 |

### 5.2 中优先级短板

| 问题 | 影响 | 建议 |
|------|------|------|
| 设施升级未实现 | 训练/医疗/球探长期投资缺口 | 加入 training / medical / scouting facility 等级 |
| 赛后新闻发布会未实现 | 士气/声望反馈少一个强交互点 | 做成轻量三选项事件，不必一开始过重 |
| 转会条款不够真实 | 大额交易策略空间有限 | 后续加入分期、浮动条款、解约金 |
| UI 长文件较多 | 后续维护成本升高 | 优先拆分 `player_detail.lua`、`market.lua`、`dashboard.lua` |

### 5.3 低优先级增强

| 功能 | 价值 | 说明 |
|------|------|------|
| 经纪人系统 | 提升谈判真实感 | 可在转会条款之后做 |
| 更多联赛 | 扩大世界 | 当前五大联赛已经足够验证循环 |
| 国际赛事扩展 | 丰富世界 | 世界杯已有，洲际赛事可后置 |
| 教程/新手引导 | 降低上手门槛 | 等核心玩法稳定后再做 |

## 七、下一步优先级建议

### P0：先补比赛引擎 v2

这是当前最重要的投资方向。`new` 的经营系统已经足够丰富，但如果比赛结果不能充分吸收阵容、体能、士气、训练、战术、角色、伤病和球员特性，其他系统的策略价值会被削弱。

建议拆成三步：

| 步骤 | 产物 | 目标 |
|------|------|------|
| 1 | `match_engine.lua` | 逐分钟事件、攻防回合、射门、犯规、牌、伤病、换人影响 |
| 2 | `tactics_resolver.lua` | 阵型、打法、球员角色、士气、体能转换为概率修正 |
| 3 | `match_report.lua` | 赛后评分、关键事件、战术反馈、球员表现摘要 |

### P1：建立最小自动化测试

不需要一开始做完整框架，但至少应覆盖以下纯逻辑：

| 模块 | 必测点 |
|------|--------|
| `transfer_manager.lua` | 报价、反报价、租借、自由球员 FSM、窗口限制 |
| `contract_manager.lua` | 到期提醒、续约概率、终止遣散费 |
| `finance_manager.lua` | 工资扣除、转会收支、财务健康等级 |
| `season_manager.lua` | 升降级、奖金、成长、退役、新赛季初始化 |
| `match_engine.lua` | 固定随机种子下的结果稳定性和关键概率边界 |

### P2：补齐设施升级与发布会

设施升级和发布会是 `FEATURE_GAP_PLAN` 中剩余最明确的功能。它们不如比赛引擎重要，但能强化“经营投入→长期收益”和“媒体回应→士气/声望变化”的反馈链。

### P3：做转会真实度扩展

在当前买卖、租借、自由签、AI 报价都可用的基础上，再加入经纪人、球员意愿、分期付款、附加条款、解约金和竞争报价，会让转会从“数值谈判”升级为“关系与风险管理”。

## 八、总结

`new` 当前已经完成了从“Lua 复刻项目”到“完整足球经理游戏雏形”的跨越。相比 `origin`，它在世界层、长期玩法和经营反馈上扩展明显，尤其是五大联赛、欧冠、世界杯、董事会、声望、求职、奖项、历史、名人堂、AI 管理和随机事件，让多赛季游玩更有目标和叙事。

但从产品质量角度看，`new` 还不能简单判定为全面超过 `origin`。`origin` 仍然在比赛引擎、类型安全、持久化成熟度和测试保障上更稳。`new` 下一阶段的关键不是继续横向堆模块，而是把比赛引擎做深，并给现有系统补上最小测试护栏。

**最终建议**：短期聚焦 `比赛引擎 v2 + 战术解析 + 比赛报告 + 最小逻辑测试`。这会把已经完成的训练、士气、阵容、转会、财务和董事会系统真正串成一个有因果、有解释、有策略回报的足球经理体验。
